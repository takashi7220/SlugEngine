## 1. 設計方針

SlugEngine は「**責務を明確に分離し、上位層が下位層に一方向にのみ依存する**」ことを根本原則とする。

各モジュールは独立してビルド・テストができる状態を保ち、特定の層の変更が他の層に波及しないよう設計する。機能を追加するときは既存の層を拡張するのではなく、必要に応じて新しい層を追加することを優先する。

エンジン全体を通じて以下の価値を優先する。

- **明示性**: 暗黙の依存や隠れた状態を持たない
- **交換可能性**: プラットフォーム実装や外部ライブラリを差し替えられる
- **検証可能性**: 各層が独立してテスト可能である

---

## 2. レイヤードアーキテクチャ

プロジェクト全体は **5 つの層** で構成される。各層は直下の層にのみ依存し、層を飛び越えた依存は持たない。

```
┌─────────────────────────────┐
│     アプリケーション層       │  エントリポイント・フレームワーク
├─────────────────────────────┤
│     ゲームエンジン層         │  ゲームロジック・コンポーネント・アセット
├─────────────────────────────┤
│     グラフィックス層         │  レンダリング・シェーダー・ハードウェア抽象
├─────────────────────────────┤
│     コア層                   │  メモリ・コンテナ・リフレクション・シリアライズ
├─────────────────────────────┤
│     数学・基盤層             │  ベクトル・行列・アロケータ
└─────────────────────────────┘
```

**上位層は下位層を知っているが、下位層は上位層を知らない。**  
たとえば `core` はグラフィックス API を一切知らず、`rhi` はゲームのアクタを知らない。

---

## 3. ライブラリの責務分離

ライブラリは「**何ができるか**」ではなく「**何に責任を持つか**」で分割する。1 つのライブラリが複数の関心事を抱えることを避ける。

### 責務の割り当て原則

- **数学・基盤層**: エンジン固有の概念を持たない純粋な計算・メモリ管理
- **コア層**: エンジンで共通して使われる仕組みを提供する。特定のグラフィックス API やゲームの概念には依存しない
- **グラフィックス層**: ハードウェアへのアクセスを抽象化し、描画に関する責務のみを持つ。ゲームロジックを知らない
- **ゲームエンジン層**: ゲームの「世界」を構築する概念（アクタ、アセット、ワールド）を定義する。ハードウェアの詳細を知らない
- **アプリケーション層**: エンジンの初期化・実行ループ・終了のみを担う

### 同一層内のさらなる分割

同一層の中でも、関心事が異なるものはサブライブラリとして分割する。グラフィックス層であれば「ハードウェア抽象」「シェーダー定義」「高レベルレンダリング」は独立したライブラリとして存在し、必要な側だけが依存を宣言する。

## 4. フォルダ構成の原則

### ライブラリ内部の構造

各ライブラリは `include/` と `src/` を分離し、公開インターフェースと実装を明確に区別する。

```
libs/<ライブラリ名>/
├── include/<ライブラリ名>/    # 公開ヘッダ（他のライブラリがインクルードする）
└── src/<ライブラリ名>/        # 非公開実装
```

`include/` 以下に置かれたヘッダはそのライブラリの公開 API である。`src/` は外部から直接インクルードしない。

### ヘッダ内のサブディレクトリ

ヘッダは **機能カテゴリ単位** でサブディレクトリに整理する。ファイル数が多い場合でも、1 つのディレクトリに詰め込まず関心ごとに分ける。  
ディレクトリ名はそのディレクトリが扱う「概念」を表す名詞とする。

### ソースツリーの汚染禁止

ビルド生成物・中間オブジェクト・IDE プロジェクトファイルはソースツリーに置かない。すべて専用の出力ディレクトリ（`_workspace/`）に集約し、ソースツリーは常にクリーンな状態に保つ。

## 5. 依存関係の規則

### 上位→下位の一方向依存

依存は常に上位層から下位層への一方向のみ許容する。循環依存は許容しない。あるモジュールが循環依存を必要とする設計になった場合、責務の分割が誤っているサインである。

### 同一層内の依存

同一層内のライブラリ間の依存は最小限に抑える。同一層のライブラリが互いに強く依存し合う場合、1 つのライブラリに統合することを検討する。

### 依存の追加判断基準

新たな依存を追加する前に以下を確認する。

1. 依存先の層は自分の層より下位か
2. その依存は本当に必要か（インターフェースで抽象化できないか）
3. 依存を追加することで循環が生じないか

### 抽象インターフェースによる依存の逆転

ハードウェアや OS など交換可能なものへの依存は、抽象インターフェース（`I` プリフィックスのクラス）を介して行う。具体的な実装クラスはインターフェースに依存し、上位層はインターフェースにのみ依存することで、実装の差し替えを可能にする。

## 6. サードパーティの扱い

外部ライブラリはすべて `thirdparty/` に隔離する。エンジン本体のコードが外部ライブラリに直接依存する箇所を最小化し、必要に応じてエンジン側のラッパーを介して利用する。

### 採用基準

外部ライブラリを採用する場合は以下の条件を満たすことを確認する。

- ライセンスがプロジェクトの利用形態と適合している
- ビルドシステムへの統合が CMake で完結できる
- 将来的に差し替えが必要になった場合に影響範囲が限定できる

### 変更禁止

サードパーティのソースコードは直接編集しない。パッチが必要な場合はフォーク管理またはラッパーによる回避を選択する。

---

## 7. コード生成の考え方

繰り返しが多く手動管理が困難なコード（シリアライズスキーマ、リフレクション情報等）は自動生成する。

### 生成コードの扱い

- 生成されたコードはソースツリーに含めてよいが、**手動編集禁止** を明示する
- 生成の入力（スキーマ定義・アノテーション）は手動で管理し、バージョン管理の対象とする
- 生成コードと手書きコードのディレクトリを明確に分離する

### 生成ツールの位置づけ

コード生成ツール自体はエンジンのランタイムに含まれず、開発時ツールとして `tools/` に配置する。ツールの言語・実装方針はエンジン本体と独立して選択してよい。

---

## 8. テストの位置づけ

テストはライブラリと対称的に構成し、テスト対象のライブラリ単位でテストプロジェクトを分ける。

### テストの粒度

- **ユニットテスト**: 単一のクラス・関数を対象とする。外部依存をできるだけ排除し、高速に実行できる状態を保つ
- **結合テスト**: 複数のライブラリにまたがる動作を確認する
- **ワークフローテスト**: エンジン全体を通した動作を実アプリケーションに近い形で検証する

### テストの独立性

各テストプロジェクトは対象ライブラリにのみリンクし、他のテストプロジェクトに依存しない。テスト間に実行順序の依存があってはならない。

---

## 9. サンプルの位置づけ

サンプルは「ライブラリの使い方を示す最小の動作例」として存在する。機能の検証にも使うが、本番品質のコードである必要はない。

### サンプルの原則

- 1 つのサンプルが示す概念は 1 つに絞る
- サンプルはエンジンの内部実装を知らなくても読み解ける粒度にする
- サンプルコードはライブラリの公開 API のみを使い、内部実装に触れない

---

## 10. ビルド出力の分離

ソースコードとビルド成果物は完全に分離する。この分離により以下を実現する。

- `git clean` 一発でリポジトリをクリーンな状態に戻せる
- 複数の構成（Debug / Release）を同時に保持できる
- CI 環境でソースツリーへの書き込みなしにビルドが完結する

ビルド成果物の出力先はプロジェクトルートからの相対パスで一元管理し、個々の CMakeLists.txt が独自のパスを設定しない。
